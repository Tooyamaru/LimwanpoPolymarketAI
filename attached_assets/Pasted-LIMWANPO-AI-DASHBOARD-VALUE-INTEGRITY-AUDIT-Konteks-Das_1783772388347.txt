LIMWANPO AI — DASHBOARD VALUE INTEGRITY AUDIT

Konteks:
Dashboard menampilkan 12 market Polymarket:
- BTC 5M / 15M / 1H
- ETH 5M / 15M / 1H
- SOL 5M / 15M / 1H
- XRP 5M / 15M / 1H

Masalah yang terlihat dari dashboard:
- UP/DOWN hampir semuanya 50.5% / 49.5%
- CONF hampir semuanya 24%
- SPREAD hampir semuanya 1.00%
- PROFIT hampir semuanya +$10.20
- Hampir semua keputusan WAIT
- Banyak nilai sama walaupun asset dan timeframe berbeda

Ini bisa saja berasal dari market real yang masih flat, tetapi harus dibuktikan.
Jangan menganggap benar hanya karena pipeline hidup.

TUGAS:
Lakukan DASHBOARD VALUE INTEGRITY AUDIT.

Tujuan:
Membuktikan asal-usul setiap angka di setiap market card dari raw source → database → engine formula → API response → frontend display.

ATURAN WAJIB:
- JANGAN ubah UI terlebih dahulu.
- JANGAN redesign.
- JANGAN tambah fitur baru.
- JANGAN tambah dummy/random/fake data.
- JANGAN hardcode nilai berbeda hanya supaya dashboard terlihat bervariasi.
- JANGAN mengubah threshold sebelum root cause terbukti.
- JANGAN menyebut data “real” tanpa menunjukkan raw evidence.
- JANGAN menyamarkan default/fallback sebagai hasil AI.
- Audit code dan data yang benar-benar sedang berjalan sekarang.

==================================================
STEP 3 — TRACE UP / DOWN PROBABILITY
==================================================

Untuk setiap market, buktikan:

- raw source UP probability
- raw source DOWN probability
- apakah berasal dari:
  - Gamma outcomePrices
  - CLOB midpoint
  - best bid/ask
  - database cache
  - formula internal
  - fallback/default

Tampilkan:

- raw value
- formula
- rounded value
- displayed value
- source timestamp

Contoh format:

BTC 5M
condition_id: ...
raw yes_bid: ...
raw yes_ask: ...
midpoint: ...
raw no value: ...
displayed UP: 50.5%
displayed DOWN: 49.5%
classification: REAL_CLOB / REAL_GAMMA / DERIVED / DEFAULT / FALLBACK / STALE / UNKNOWN

Jika semua 12 market sama:
- buktikan raw source memang sama.
- jika karena fallback/default, tandai INVALID_DEFAULT atau INVALID_FALLBACK.
- jika karena stale shared cache atau wrong market mapping, tandai BUG.

==================================================
STEP 4 — TRACE SPREAD
==================================================

Untuk setiap market tampilkan:

- yes_bid
- yes_ask
- no_bid
- no_ask jika tersedia/derived
- raw spread
- spread formula
- displayed spread
- timestamp

Audit apakah 1.00% berasal dari:
- bid/ask real
- clamp
- fixed minimum
- default
- rounding
- shared value
- missing orderbook fallback

Jika semua selalu 1.00%, cari:

- hardcoded 0.01
- default spread
- min/max clamp
- frontend fallback
- stale orderbook reused across cards

==================================================
STEP 5 — TRACE CONFIDENCE
==================================================

Untuk setiap market tampilkan semua input confidence:

- signal confidence
- opportunity score
- strategy score
- market quality
- liquidity/depth factor
- spread factor
- volatility
- momentum
- trend
- conflict penalty
- risk adjustment
- dynamic weight contribution
- final raw confidence
- rounded/displayed confidence

Tampilkan formula lengkap dan angka aktual.

Jika semua tampil 24%:
- cek apakah 24 berasal dari fixed AMM-init path
- default confidence
- fallback score
- constant opportunity score
- missing signal_confidence
- rounding
- shared object/reference bug

Klasifikasikan:
- REAL_COMPUTED
- DERIVED_FROM_REAL_DATA
- DEFAULT
- FALLBACK
- CONSTANT
- STALE
- BUG
- UNKNOWN

Tidak boleh ada UNKNOWN pada hasil akhir.

==================================================
STEP 13 — REMEDIATION RULES
==================================================

Jika ditemukan bug:
- perbaiki minimal.
- jangan redesign.

Jika ditemukan fallback/default:
- jangan sembunyikan.
- gunakan null / NOT_AVAILABLE / INSUFFICIENT_DATA jika tidak ada data real.
- UI harus menampilkan N/A atau data unavailable, bukan angka seolah-olah real.

Jika nilai sama karena rounding:
- tampilkan raw precision di audit.
- jangan ubah UI kecuali rounding membuat data berbeda terlihat identik secara menyesatkan.

Jika market benar-benar flat:
- jangan paksa variasi.
- dokumentasikan raw evidence bahwa tiap market punya condition_id/token/orderbook berbeda tetapi nilai kebetulan sama.

Jika profit adalah fixed simulation amount:
- perbaiki label menjadi POTENTIAL PAYOUT / SIMULATED RETURN jika itu arti sebenarnya.
- jangan menyebut PROFIT jika belum realized atau belum valid expected value.

Gunakan PARTIAL jika:
- audit selesai tapi beberapa nilai belum dapat ditelusuri

Gunakan FAIL jika:
- angka palsu/default ditampilkan sebagai data real
- banyak card memakai source row yang sama secara salah
